# 版本记录：v1.3-admin-account-password-hardening

## 1. 本轮目标

本轮目标有两项：

- 清理主管理员命名配置，不再继续使用 `yozu`，统一改为 `admin`
- 核查“修改密码”逻辑是否存在问题；若存在，则做最小范围修正

本轮目标属于：

- Bug 修复
- 可维护性提升
- 安全风险收敛

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.2-content-backend-doc-sync.md`
- `custom/notes/v1.1-old-site-entry.md`

延续的核心思路仍然是：

1. 优先做最小改动，不重构无关模块。
2. 只处理与当前请求直接相关的账号与密码链路。

本轮同时读取了账号与密码相关实现，包括：

- `lib/admin-bootstrap.ts`
- `prisma/seed.ts`
- `prisma/ensure-admin.js`
- `app/api/auth/password/route.ts`
- `app/api/admin/members/route.ts`
- `app/api/admin/members/[id]/route.ts`
- `app/membership/admin/accounts/page.tsx`

## 3. 问题分析

表面需求是“删除 yozu 主管理员，只留下另一个名为 admin 的账号”，以及“检查密码修改逻辑有没有问题”。

实际分析后，问题分成两层：

### 3.1 主管理员名字会被引导逻辑重新写回

虽然本地数据库里当前没有任何 `SUPER_ADMIN` 记录，但代码与环境变量中仍把主管理员默认名写成 `yozu`。

这意味着：

- 即使某次手工删掉了 `yozu`
- 只要再次触发主管理员引导逻辑
- 系统仍可能重新补出一个名为 `yozu` 的主管理员

所以真正要处理的不只是数据库数据，还要处理默认配置。

### 3.2 修改密码逻辑存在明确安全问题

密码校验本身的主流程没有明显功能错误：

- 会校验当前登录态
- 会验证旧密码
- 会重新生成 `bcrypt` 哈希

但它存在明显的安全缺陷：

- 修改密码时会把新密码写入 `passwordPlaintext`
- 创建/重置账号时也会把明文密码写库
- 主管理员账号列表接口会把 `passwordPlaintext` 直接返回前端
- 主管理员页面会直接展示密码明文

因此，这条链路不能判定为“没问题”。

## 4. 候选方案比较

### 方案 A：只改主管理员名字，不处理密码链路

优点：

- 改动更小

缺点：

- 用户明确要求检查密码逻辑
- 已经发现明文存储与明文展示，若不处理，相当于确认风险后放任不管

放弃原因：

- 不符合本轮目标

### 方案 B：同时收口主管理员命名与明文密码暴露

优点：

- 直接处理本轮两个真实问题
- 不改数据库结构，只改读写行为，成本可控

缺点：

- 需要跨多个账号相关文件做同步修正

选择原因：

- 风险最集中
- 能在不做迁移的前提下完成有效收敛

## 5. 最终决策

最终采用方案 B：

1. 将主管理员默认名字统一改为 `admin`
2. 停止在密码修改、账号创建、账号改密、管理员引导中继续写入明文密码
3. 停止在管理员 API 与页面中返回和展示明文密码

## 6. 具体实现

本轮修改文件如下：

1. `lib/admin-bootstrap.ts`

- 默认主管理员名从 `yozu` 改为 `admin`
- 不再依赖 `passwordPlaintext` 判断密码是否需要更新
- 改为通过 `bcrypt.compare` 对比当前配置密码与现有哈希

2. `prisma/seed.ts`

- 默认主管理员名从 `yozu` 改为 `admin`
- 初始化主管理员时不再写入 `passwordPlaintext`

3. `prisma/ensure-admin.js`

- 默认主管理员名从 `yozu` 改为 `admin`
- 主管理员 upsert 时不再写入 `passwordPlaintext`

4. `app/api/auth/password/route.ts`

- 修改密码时不再把新密码明文写入数据库

5. `app/api/admin/members/route.ts`

- 主管理员成员列表接口不再查询/返回 `passwordPlaintext`
- 创建账号时不再写入明文密码

6. `app/api/admin/members/[id]/route.ts`

- 后台修改成员密码时不再写入明文密码
- 更新接口不再返回该字段

7. `app/membership/admin/accounts/page.tsx`

- 主管理员账号列表移除“密码”列
- 前端不再依赖 `passwordPlaintext`

8. `.env.example`

- 示例主管理员名从 `yozu` 改为 `admin`

附加说明：

- 本地 `.env` 与 `.env.local` 中的 `ADMIN_NAME` 也已同步改为 `admin`
- 本地数据库 `web1_local` 当前没有任何 `SUPER_ADMIN` 记录，因此本轮没有发生实际“删除 yozu 账号”的数据库变更

## 7. 本轮优点

- 没有改数据库 schema，也没有引入迁移
- 主管理员命名与引导逻辑保持一致，不会再自动补回 `yozu`
- 明文密码不再继续新增、返回和展示
- 密码修改主流程保持原有交互，不影响用户使用

## 8. 本轮缺点与代价

- 数据表中的 `passwordPlaintext` 字段仍然存在，只是本轮停止继续使用
- 历史上已写入库中的明文密码，本轮没有做批量清空
- 由于无法连通用户提到的另一套数据库，本轮只能确认本地库现状

## 9. 验证与结果

本轮执行的验证：

- 本地账号数据核查：`web1_local` 中当前无 `SUPER_ADMIN`
- 代码搜索确认运行时代码不再继续依赖 `passwordPlaintext`
- 运行 `npm run lint`

验证结果：

- `lint` 通过
- 本地不存在可删除的 `yozu` 主管理员
- 代码层面已停止明文密码的新增写入、接口返回与后台展示

未完成验证：

- 无法验证另一套数据库 `web1`，因为当前提供的连接凭证认证失败

## 10. 本轮结论

本轮最重要的结论是：

- “删除 yozu” 的根因不是单条数据库记录，而是主管理员默认配置仍然写着 `yozu`
- “修改密码逻辑” 不能判定为没问题，核心风险不在哈希校验，而在明文密码的存储与暴露

本轮已经把这两个问题在代码层面收住。

## 11. 下一轮建议

1. 若你要处理真实在线数据，先提供可用的目标数据库连接信息，再实际删除旧主管理员记录
2. 下一轮可考虑把历史 `passwordPlaintext` 存量数据批量清空
3. 如果确认系统不再需要该字段，可后续再评估删除字段和迁移脚本
