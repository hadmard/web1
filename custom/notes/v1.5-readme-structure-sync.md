# 版本记录：v1.5-readme-structure-sync

## 1. 本轮目标

本轮目标是基于当前仓库真实结构，更新根目录 README，使其能够准确反映：

- 这个项目现在到底包含哪些模块
- 主导航与兼容路由之间的关系
- 会员系统与管理员后台的范围
- 当前真正可用的启动、数据库与文档入口

本轮目标属于：

- 可维护性提升
- 文档同步
- 项目交接基础建设

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.4-cover-visibility-and-site-check.md`
- `custom/notes/v1.2-content-backend-doc-sync.md`

读取这两期记录后，可以延续两条和本轮最相关的结论：

1. 继续坚持最小改动原则，不因为文档任务顺手改业务代码。

## 3. 问题分析

当前 README 的主要问题不是“完全错误”，而是明显滞后于仓库结构演化。

表面现象包括：

- README 仍把项目描述成较早期的内容站概览
- 没有体现会员系统和管理员后台的实际范围
- 没有体现 `app/api/` 已经形成较完整的路由层
- 仍然引用 `docs/COMPONENTS.md`、`docs/DEPLOYMENT.md`，但当前仓库中并不存在这些文件
- 没有说明 `/market`、`/data`、`/gallery` 与当前主导航之间的实际关系

真正的核心矛盾是：

- 仓库结构已经扩展成“公开前台 + 会员系统 + 管理后台 + 数据层”
- README 仍停留在“公开栏目页 + 基础启动说明”的层级

如果继续沿用旧 README，后续接手者会在两个方面产生误判：

1. 低估项目边界，以为这里只是前台站点。
2. 高估某些过期文档入口的可用性，以为还有 `docs/` 目录可参考。

## 4. 候选方案比较

### 方案 A：只做小修小补

核心思路：

- 保留原 README 主体结构
- 只补几条命令或目录说明
- 不重写栏目、会员和后台说明

优点：

- 改动更小
- 成本更低

缺点：

- 无法解决 README 与当前项目结构脱节的问题
- 仍然会留下“项目边界不清”的理解偏差
- 旧的无效文档链接问题也不会被真正处理干净

放弃原因：

- 只能修表面，不能修核心信息失真

### 方案 B：在不改代码的前提下，按真实结构重写 README

核心思路：

- 保留 README 作为单文件入口
- 基于当前实际目录与关键文件，重组 README 的章节
- 补充主模块、API、数据库、目录结构、环境变量、初始化顺序
- 明确指出兼容路由与失效文档入口

优点：

- 能直接反映当前项目真实边界
- 仍然只改文档，不触碰业务实现
- 后续接手者可以更快定位模块

缺点：

- README 体量会比旧版更长
- 仍然是静态快照，后续结构变化时需要继续维护

选择原因：

- 在“只改文档”的前提下，信息收益最高
- 最符合本轮“看结构后更新 README”的任务目标

## 5. 最终决策

最终采用方案 B。

关键决策如下：

1. 不改任何业务代码，只更新 README 与记录文件。
2. README 以当前实际主链路为准，不把兼容重定向页写成正式主栏目。
3. README 中明确加入会员系统、管理员后台与 API 层说明。
4. 移除旧 README 中已经失效的 `docs/` 引用，改为当前仍存在的说明入口。
5. 把初始化顺序写清，降低首次接手时的试错成本。

## 7. 本轮优点

- README 与当前仓库结构重新对齐
- 会员系统和管理员后台不再被遗漏
- 首次接手者能更快理解项目边界与启动步骤
- 去除了已经失效的文档引用，减少误导
- 改动完全停留在文档层，没有引入业务风险

## 8. 本轮缺点与代价

- README 仍然是人工维护的结构快照，后续目录变化后仍可能再次过期
- 本轮没有顺手拆出更细的专项文档，例如权限说明、部署说明、内容模型映射说明
- 对各 API 的行为只做了分组概览，没有逐接口说明

## 9. 验证与结果

本轮完成了以下验证：

- 读取并核对 `app/`、`components/`、`lib/`、`prisma/`、`custom/` 的实际目录
- 读取 `package.json`、`.env.example`、`prisma/schema.prisma`、`prisma/README.md`
- 核对会员系统、管理员后台、兼容路由与 API 分组的真实文件位置
- 执行 `git diff --check` 做最小文本校验

当前确认结论：

- 旧 README 确实滞后于当前项目结构
- 当前仓库中不存在旧 README 引用的 `docs/` 文档
- 当前项目已经具备公开前台、会员系统、管理后台与 API 层的完整分层

## 10. 本轮结论

本轮最重要的结论是：

这个仓库已经不是“只有公开内容页的 Next.js 站点”，而是一个有明确后台与共建链路的内容平台。README 若继续停留在初始版本写法，会持续误导后续开发判断。

本轮确认了两点应默认继承的经验：

1. README 应优先描述当前真实结构，而不是历史阶段的理想化分类。
2. 兼容链接、重定向页和失效文档入口，必须在说明中主动标明，不能让接手者自行猜测。

## 11. 下一轮建议

1. 若后续继续补文档，优先拆一份“会员与管理员后台操作说明”
2. 若权限逻辑继续演化，可单独补一份“角色与权限矩阵”文档
3. 若部署方式稳定下来，再补独立的部署说明，而不是继续把部署细节堆进 README
