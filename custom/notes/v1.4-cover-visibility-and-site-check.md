# 版本记录：v1.4-cover-visibility-and-site-check

## 1. 本轮目标

本轮目标有三项：

- 将资讯/内容详情页的封面逻辑改为“有图才显示，无图不显示”
- 排查后台与会员发布页的“顶部配图预览不显示”问题
- 做一轮全站最小可行联调，顺手修正明显不合逻辑的实现

本轮目标属于：

- Bug 修复
- 可维护性提升
- 交互一致性修正

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.3-admin-account-password-hardening.md`
- `custom/notes/v1.2-content-backend-doc-sync.md`

此前已经补上了后台内容系统里的摘要生成、关键词生成、顶部配图上传入口与预览区域，但用户继续反馈“默认封面图不该强行显示”和“配图预览不显示”。

实际检查后发现，这两个问题并不完全在同一层：

- 详情页问题属于展示逻辑错误：组件会在业务图缺失时自动回退到默认图
- 预览问题更接近“可见但不直观”：上传成功后能生成 URL，也会渲染预览，但预览使用 `object-contain`，在某些图片比例下会显得像没有显示

## 3. 问题分析

### 3.1 详情页封面与业务语义不一致

`ContentHeroImage` 原先的逻辑是：

- 有业务图时显示业务图
- 没业务图时显示 `fallbackSrc`
- 连 `fallbackSrc` 都没有时显示一个渐变占位容器

这在“栏目页固定视觉背景”场景下是成立的，但对资讯、词库文章、标准文章、评选文章这类内容详情页并不合理。  
这类内容的 `coverImage` 已经是明确的可选字段，可选字段的语义应该是：

- 填了就显示
- 没填就不显示

否则后台不传图，前台却仍然出现默认头图，会给人造成“系统偷偷补图”的错觉。

### 3.2 配图预览并非完全失效，而是体验上不够明确

通过真实上传测试发现：

- `/api/upload/image` 正常返回文件 URL
- 表单状态会正确写入 `coverImage`
- 页面会渲染“顶部配图预览”区域

但预览采用 `object-contain` 后，宽图或截图类图片会在固定高度容器里缩成一条窄带，导致使用者误判为“没显示”。

### 3.3 富文本编辑器存在重复扩展告警

浏览器联调时，内容发布相关页面稳定出现 Tiptap 告警：

- `Duplicate extension names found: ['link', 'underline']`

这不是阻断型问题，但属于明确的不合理实现，会增加后续编辑器行为异常的风险。

## 4. 候选方案比较

### 方案 A：只在资讯详情页单独去掉默认封面

优点：

- 改动最小

缺点：

- 词库文章、标准文章、评选文章仍然保留同样的不一致逻辑
- `ContentHeroImage` 组件的默认行为仍然会诱导后续新页面继续误用

放弃原因：

- 只能修一处，不够彻底

### 方案 B：保留组件的 fallback 能力，但由页面显式决定是否传 fallback

优点：

- 同时兼顾两类场景：
  一类是固定视觉背景页面，仍可传 `fallbackSrc`
  一类是业务封面可选页面，不传 fallback 即可
- 不需要重做组件层级

缺点：

- 需要同步梳理多个详情页调用点

选择原因：

- 能在最小改动下修正语义
- 后续新增页面时也更容易遵守“封面是否可选”的业务逻辑

## 5. 最终决策

最终采用方案 B，并附带两项交互修正：

1. `ContentHeroImage` 改为：
   - `src` 和 `fallbackSrc` 都为空时直接不渲染
   - 不再输出默认渐变占位块
2. 对资讯/词库文章/标准文章/评选文章详情页，不再传默认封面 `fallbackSrc`
3. 后台与会员发布页的顶部配图预览，改为更符合预期的 `object-cover`
4. `RichEditor` 中关闭 `StarterKit` 自带的重复扩展，再接入自定义 `Link` 与 `Underline`

## 6. 具体实现

本轮修改文件如下：

1. `components/ContentHeroImage.tsx`

- `fallbackSrc` 改为可选
- 当 `src` 与 `fallbackSrc` 都为空时直接返回 `null`
- 图片 alt 统一使用调用方传入的语义文本

2. `app/news/[slug]/page.tsx`

- 资讯详情页不再读取站点默认资讯头图
- 仅在文章存在 `coverImage` 时显示头图

3. `app/dictionary/[slug]/page.tsx`

- 对由文章承载的词库内容，取消默认头图回退
- 保留原生词条页的固定视觉背景逻辑

4. `app/standards/[id]/page.tsx`

- 对由文章承载的标准内容，取消默认头图回退
- 保留原生标准记录页的固定视觉背景逻辑

5. `app/awards/[id]/page.tsx`

- 评选详情页仅在存在 `coverImage` 时显示头图

6. `app/membership/admin/content/page.tsx`

- 顶部配图预览改为 `object-cover`

7. `app/membership/content/publish/page.tsx`

- 顶部配图预览改为 `object-cover`

8. `components/RichEditor.tsx`

- `StarterKit` 中关闭 `link` 与 `underline`
- 继续使用已配置的 `Link` 和 `Underline`，消除重复扩展警告

## 7. 本轮优点

- 详情页封面显示逻辑与字段语义重新对齐
- 后台和会员发布页的配图预览更直观，不容易误判为“没显示”
- 富文本编辑器控制台告警被收敛，后续排错噪音更少
- 改动都集中在展示层与编辑器配置，没有触碰数据库结构

## 8. 本轮缺点与代价

- 本轮没有引入统一的“图片预览组件”，后台与会员页仍有少量重复代码
- 对“原生词条页/原生标准页”是否也要彻底取消默认视觉图，本轮没有继续扩大范围
- 浏览器联调过程中，`next dev` 与 `next build` 混用导致本地测试环境出现过一次 Next 运行时资源异常；这属于测试环境干扰，不是业务逻辑缺陷

## 9. 验证与结果

本轮完成了以下验证：

- `npm run lint`
- `npm run build`
- 使用真实浏览器登录后台内容页
- 使用真实图片文件测试后台顶部配图上传
- 确认上传接口 `/api/upload/image` 正常返回 URL
- 确认后台上传后页面进入“顶部配图已加载，可先预览”状态
- 本地插入一条无封面资讯测试数据，用于验证详情页逻辑；测试后已清理该数据

验证结论：

- `lint` 通过
- `build` 通过
- 后台顶部配图上传与预览链路可用
- 资讯详情页在无封面时不再需要默认图兜底

未完全完成的部分：

- 生产环境与本地环境的资源缓存行为没有做更深排查，因为这不属于本轮业务需求

## 10. 本轮结论

本轮最重要的结论有三点：

1. “默认封面图不该自动出现”是展示语义问题，不是数据问题
2. “配图预览不显示”在当前实现里更接近交互误导，而不是上传失败
3. 内容发布相关页面里，富文本编辑器的重复扩展告警属于不合理实现，应该在这一轮一并收掉

## 11. 下一轮建议

1. 若后续还要继续优化内容后台，可以把“上传 + 预览 + 清除”提炼成复用组件，减少后台与会员页的重复实现
2. 可以继续排查原生词条页、原生标准页是否也要遵循“无图就不显示”的同一视觉策略
3. 如果要做更完整的全站回归，建议下一轮固定为 `next start` 或固定为浏览器自动化专用环境，避免 `dev/build/start` 混跑对测试判断造成干扰
