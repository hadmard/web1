# 版本记录：v1.2-content-backend-doc-sync

## 1. 本轮目标

本轮目标是根据外部文稿 `D:\desktop\文字文稿1(3).docx` 中的文字说明与截图，对网站后台内容系统做一轮对齐修正。

本轮优先处理的问题集中在整木资讯后台，但要求要能同步覆盖同类栏目与同类表单能力，具体包括：

- 摘要可根据当前内容自动生成，且仍允许人工修改
- 关键词支持自动生成，且仍允许人工修改
- 上传顶部配图后应立即出现可视预览
- 内容管理中的标题可直接打开前台详情页，且在新标签页打开
- 点击“修改”后要有明确可见的跳转反馈，不能表现为“没反应”
- 资讯详情页头图区域不要再把文章标题当成图片替代文本展示出来

本轮目标属于：

- 功能补充
- Bug 修复
- 可维护性提升

## 2. 背景与上下文

本轮承接以下历史记录：

- `custom/notes/v1.1-old-site-entry.md`
- `custom/notes/v1.0-commercial-baseline.md`

与当前任务最相关的继承结论有两条：

1. 继续坚持最小改动原则，优先复用现有内容系统结构，不为单个需求重做整套后台架构。
2. 对前台可见文案与视觉表现做正式化替换时，应优先从真实使用场景出发，而不是再堆开发期占位行为。

本轮没有在 `custom/notes/experiments/` 中发现与该文稿直接对应的历史实验记录，因此本次判断主要依据当前代码现状与文稿截图。

修改前，后台已经具备：

- `RichEditor` 中已有撤销 / 重做按钮
- 标签已有自动识别逻辑
- 管理员内容系统与会员发布系统各有一套表单实现

但文稿中的问题说明表明，现有交互仍然不足以让使用者感知到这些能力，或者关键链路上仍存在缺口。

## 3. 问题分析

文稿里表面上提了多条需求，但实际可以归并为三类核心矛盾：

### 3.1 自动化能力“有底层、没前台体验”

现有系统已经有标签自动识别与编辑器撤销功能，但：

- 标签按钮文案偏“识别”，不够贴近“关键词自动生成”的使用心智
- 摘要没有对应的快捷生成入口
- 撤销虽然存在，但本轮无需再新增底层能力

真正的问题不是完全缺少算法，而是入口表达和交互闭环不完整。

### 3.2 图片上传链路“能存值、不能确认结果”

现有顶部图片字段支持输入 URL 或上传文件，但上传后只回填 URL，不给预览。

这会造成两个直接后果：

- 使用者无法确认上传是否成功
- 使用者无法理解这个字段最终会出现在前台哪个位置

### 3.3 管理链路“功能存在，但反馈不明显”

内容管理页原本的“修改”是展开同页底部的编辑表单，不会新开页面，也不会主动滚动到编辑区。
在列表较长的情况下，用户点击后极容易误判为“没有反应”。

同时，标题只是纯文本，无法直接进入对应前台详情页，不符合文稿里“成功发布后可直接点标题查看”的要求。

## 4. 候选方案比较

### 方案 A：按文稿字面要求新增独立“修改页面”与专门摘要生成接口

核心思路：

- 新建单独编辑路由
- 新建新的摘要生成接口或服务层
- 为资讯栏目单独定制一套后台体验

优点：

- 更贴近“跳转到修改页面”的字面描述
- 后续可以继续扩展更多编辑流程

缺点：

- 改动面明显扩大
- 会破坏当前单页式内容系统结构
- 需要额外处理管理员与会员两套页面的一致性

风险：

- 容易把本轮“交互补齐”做成“后台重构”
- 可能引入新的权限与路由状态问题

放弃原因：

- 与当前仓库“尽量局部修改”的原则不一致
- 当前问题本质是可见反馈不足，不一定需要新增页面才能解决

### 方案 B：保留现有单页结构，只补足自动生成、预览与跳转反馈

核心思路：

- 沿用现有管理员页与会员页结构
- 用现有文本能力生成摘要
- 用现有标签识别能力承接“关键词自动生成”
- 在点击修改后自动滚动到编辑区
- 在内容管理列表中给标题补前台预览链接

优点：

- 改动集中，风险低
- 不需要改数据库，也不需要改权限模型
- 能同时覆盖管理员页与会员页

缺点：

- “修改页面”仍然是同页编辑区，而不是新路由
- 摘要生成采用规则提炼，不是 AI 生成

风险：

- 如果后续用户明确要求真正的独立编辑页，本轮方案仍可能需要继续演进

选择原因：

- 最符合当前结构
- 能直接响应文稿中的主要体验问题
- 实现成本与收益比最好

## 5. 最终决策

最终采用方案 B，并做以下关键决定：

1. 不新建后台编辑路由，继续使用现有单页式内容系统。
2. 点击“修改”后通过自动滚动到编辑区的方式，补足“可见跳转反馈”。
3. 摘要自动生成使用现有文本清洗与截断能力，先实现稳定可控的规则生成。
4. 关键词自动生成继续沿用现有标签建议逻辑，但在界面上改为“关键词”表述。
5. 顶部配图预览使用原生 `img`，避免因任意上传地址触发 `next/image` 的远程域名限制。
6. 资讯详情页的头图改为装饰性替代文本，避免图片异常时把标题再次显示在图框里。

## 7. 本轮优点

- 没有改数据库结构，也没有改现有权限体系
- 同一轮同时覆盖了管理员后台与会员发布后台，避免再次出现双轨体验不一致
- 摘要、关键词、图片预览三个入口都形成了“生成/上传后立即可见”的闭环
- 标题直达前台详情页，缩短了后台核对发布结果的路径
- “修改没反应”的问题通过自动滚动得到直接修复
- 前台资讯详情页的头图异常显示更收敛，不再把标题叠加成重复信息

## 8. 本轮缺点与代价

- 摘要自动生成目前仍是基于文本截断的规则生成，不是更强语义级摘要
- “修改页面”本质上仍是同页编辑区，不是真正独立路由
- 关键词生成沿用现有标签规则，命中质量仍受关键词词表覆盖度影响
- 顶部配图预览使用原生 `img`，虽然更稳，但不享受 `next/image` 的优化能力
- 这次只修了资讯详情页头图替代文本，没有顺手扩展到其它内容详情页

## 9. 验证与结果

本轮执行的验证：

- 代码级 diff 复核，确认改动集中在文稿对应范围
- 运行 `npm run lint`

验证结果：

- `lint` 通过，无警告无错误
- 代码层面已具备摘要自动生成、关键词自动生成、顶部配图预览、标题新标签预览、修改自动滚动等能力

未执行验证：

- 未进行完整浏览器联调
- 未逐条人工点击管理员后台与会员后台所有栏目进行可视回归

原因：

- 本轮优先先完成代码级修复与静态验证

剩余风险：

- 若线上上传图片返回的 URL 规则与本地环境不同，仍需要实际页面联调确认预览显示
- 若用户最终坚持“必须是独立修改页”，则还需要下一轮继续演进路由方案

## 10. 本轮结论

本轮最重要的结论是：

文稿中提到的多数问题并不是“底层完全没有能力”，而是“当前交互没有把已有能力有效暴露出来”。因此最合适的修法不是重做后台，而是在现有内容系统上补清晰入口、即时反馈和前台跳转链路。

本轮证实了以下判断：

- 摘要生成可以先用规则方案快速落地
- 关键词自动生成可以复用现有标签规则
- “修改没反应”通过自动滚动即可显著改善，不必马上新建独立编辑页

本轮修正了一个旧判断：

- 原先把图片字段单纯命名为“封面图片”对使用者不够清晰，文稿中的截图更接近“顶部配图”理解

## 11. 下一轮建议

1. 实际登录后台做一次管理员页与会员页联调

- 收益最高
- 能直接确认图片预览、摘要生成、标题跳转、修改滚动是否符合预期

- 风险中等
- 需要在确认本轮方案仍不满足后再做

3. 评估是否给其它详情页同步去掉标题型替代文本

- 风险最低
- 可顺着本轮资讯页的修法继续统一
